# 执行上下文

js是一门单线程的解释性语言,是按顺序一段一段执行代码的，每当执行一段代码是都会进行一个准备工作，也就是常说的预编译阶段；
经常听到的所谓执行上下文是维护执行当前代码所需要的变量声明、this等，一堆概念肯定记不住，只要知道它是`运行js代码的环境即可,代码的运行必定会创建一个执行上下文，执行上下文也分了几种

  1. 全局执行上下文
    全局执行上下文只有一个，在js程序开始执行时由js引擎创建，并且会执行所有在顶层的js代码; 什么时候产生：当浏览器接收到一段可执行代码前，会准备执行JavaScript时所需要的一些基础环境，其中就包括全局执行上下文。
  2. 函数执行上下文
    函数执行时创建上下文，每当一个函数执行都会创建一个相对独立的上下文，同一个函数被多次调用也会创建不同的上下文；
  3. eval上下文
    使用eval函数时创建的上下文，基本不会遇到；

# 执行栈

既然每次调用函数都会创建一个执行上下文，那么这些上下文都是需要管理的，执行栈可以说是管理这些执行上下文的地方，每当创建一个执行上下文都会被压入栈顶执行，js引擎只会执行此时在栈顶的上下文代码，顶部的执行上下文也被称为活跃执行上下文，执行完成后将该上下文在栈内移除，继续执行下一个在顶部的上下文,也就是后续代码，始终遵循着先入后出的一种原则；在js执行期间，在栈底一定会存在一个全局执行上下文，因为所有的函数上下文一定都是在全局代码执行期间产生的；

此时 调用 factorial 函数会创建一个执行上下文，内部再次调用了本身再次创建一个执行上下文，一直调用 n 次，即
[global] ==> [global,factorial] ==> [global,factorial,factorial] ==> [global,factorial，factorial，factorial，factorial,...]
如果调用执行n次则执行上下文里会被压如 n 个执行上下文，因为由于函数代码并没有执行所以并不会弹出栈销毁；一但超出内存大小，则会爆栈，程序崩溃，比如 factorial(100000000),递归虽然好用，但得谨慎；

这就是执行栈，用来管理执行上下文的一种先入后出的栈结构,代码的运行必定会创建一个执行上下文,将执行上下文压入栈顶，执行完成后在从栈中弹出，这便是js代码一段一段执行的工作原理；

# 执行过程

上面只是对执行上下文有个大致的了解，但执行上下文远不止这些东西,后面将从ES3和ES6来对执行上下文有个更深层次的了解
执行上下文的创建分两个阶段

  1. 创建阶段
    准备执行代码时需要的环境
  2. 执行阶段
    就单纯的执行代码

# 创建阶段

# ES3

在ES3及之前，对于每个执行上下文，创建阶段都会初始化三个重要属性

  1. 变量对象
  2. 作用域链
  3. this绑定
    在全局执行上下文中，this 的值指向全局对象。在函数执行上下文中，this 的值取决于该函数是如何被调用的。如果它被一个引用对象调用，那么 this 会被设置成那个对象，否则 this 的值被设置为全局对象或者 undefined（在严格模式下）;

这三个东西后面会一一讲到

# 变量对象(VO/AO)

变量对象(VO)存储了在上下文中定义的变量和函数声明，全局上下文中的变量对象就是全局对象(window)； 在函数上下文中，函数执行上下文中的变量对象叫做活动对象(AO),也就是说函数上下文中的活动对象(AO)就是变量对象，它是在进入函行上下文的时候被初始化激活的，同样存储着函数内部定义的变量和内部函数声明，活动对象是在进入函数上下文时刻被创建的，通过函数arguments 属性初始化，也就是函数的参数；

上面提到了一下全局对象, W3C对全局对象的一些介绍: >全局对象是预定义的对象，作为 JavaScript 的全局函数和全局属性的占位符。通过使用全局对象，可以访问所有其他所有预定义的对象、函数和属性。
在顶层 JavaScript 代码中，可以用关键字 this 引用全局对象。因为全局对象是作用域链的头，这意味着所有非限定性的变量和函数名都会作为该对象的属性来查询。
例如，当JavaScript 代码引用 parseInt() 函数时，它引用的是全局对象的 parseInt 属性。全局对象是作用域链的头，还意味着在顶层 JavaScript 代码中声明的所有变量都将成为全局对象的属性。

这些介绍自行品会~~~~，先瞅瞅js变量的生命周期

变量生命周期
当 js 引擎处理变量时，会经历如下几个周期：

  1. 声明阶段：是在作用域中注册一个变量。
  2. 初始化阶段：是分配内存并与作用域中的变量创建绑定，变量将使用 undefined 自动初始化。
  3. 赋值阶段：为初始化的变量赋值。

声明阶段的变量不能被调用，否则会抛出异常

上面说了声明的变量都是存储在变量对象中的，那么它是什么时候存储及怎样存储的？
执行上下文是运行代码时的环境,执行上下文的代码在都会经两个过程：分析和执行过程

-分析(也叫进入执行上下文)
在执行代码之前，会预先对代码进行一次分析，此时代码还并没有执行，会初始化变量对象，得到：
1. 将函数形参和实参映射作为变量对象的属性(如果是函数上下文)，没有实参属性值为undefined。
2. 将所有函数声明作为变量对象的属性
3. 将所有的变量声明作为变量对象的属性，如果是var声明的变量，会被初始化(也就是上面变量声明周期的第二个阶段)。let,const声明的变量则会被保持声明阶段（其实ES3 还没有这两个，这里先提一下，后面词法环境，变量环境也是差不多）

此时变量对象会包含代码该执行上下文代码所有所有变量声明，函数声明。后面到执行代码阶段时，变量会从变量对象里面去查找，也会根据代码，修改变量对象的值；

# 执行阶段

执行
也就是js引擎根据顺序执行代码，会根据代码来修改变量对象中变量的值。

现在再来看变量提升

# 变量提升

# 作用域链

作用域
一般的编程语言在编译源代码时都会有三个步骤：

  1. 词法分析 将源代码分解成词法单元,比如 var a = '二水' ，分解成 关键字 var,标识符 a，赋值操作 =,值 '二水'
  2. 语法分析 根据词法单元生成AST抽象语法树, 可以去这里 AST在线调试工具 (opens new window) 瞅瞅什么是AST
  3. 代码生成 将AST转换为可执行的代码,作用域也是在这一过程中确定

作用域是指代码中定义变量的一个独立的区域，不同作用域下同名变量也不会有冲突。作用域规定了如何查找变量，也就是规定当前执行代码对变量的访问权限，能不能访问到该变量。
作用域分两种:

  1. 静态作用域(词法作用域)
    此类作用域在代码定义的时候就确定了，依赖于与代码的书写位置
  2. 动态作用域
    此类作用域根据代码的调用位置决定，与代码书写位置无关

js采用的是静态作用域，也就是作用域在代码书写的时候就已经确定了

    var value = 1;

    function foo() {
        console.log(value);
    }

    function bar() {
        var value = 2;
        foo();
    }

    bar(); // 输出 ？

根据一段例子来区别两种作用域
假设采用静态作用域，分析下执行过程：

执行 foo 函数，先从 foo 函数内部查找是否有局部变量 value，如果没有，就根据书写的位置，查找上面一层的代码，也就是 value 等于 1，所以结果会打印 1。

假设采用动态作用域，分析下执行过程：

执行 foo 函数，依然是从 foo 函数内部查找是否有局部变量 value。如果没有，就从调用函数的作用域，也就是 bar 函数内部查找 value 变量，所以结果会打印 2。

前面我们已经说了，JavaScript采用的是静态作用域，所以这个例子的结果是 1。
目前js里有三种作用域，这三种作用域想必都知道，就不多说。

  1. 全局作用域
  2. 函数作用域
  3. 块级作用域(ES6)

作用域链

前边变量对象时已经说过，当查找变量的时候，会先从当前上下文的变量对象中查找(js查找变量的方式有两种,LHS和RHS，有兴趣自己了解)，如果没有找到，就会从父级执行上下文的变量对象中查找，一直找到全局上下文的变量对象，也就是全局对象。这样由多个执行上下文的变量对象构成的链表就叫做作用域链；
一个函数的作用域在其声明时就已确定，因为每个函数都有一个内部属性 [[Scope]]，在函数创建的时候，就会将父级的变量对象的保存到其中,也就是说当前函数的[[Scope]]是在其父级预编译时确定的，保存了父级的作用域链。
注意的是[[Scope]]并不代表完整的作用域链，此时[[Scope]]里只有父级的作用域链，只有当函数本身的AO创建完毕后将AO合并到[[Scope]]的前端 scope = [AO].concat([[Scope]]),scope才算是一个作用域链的创建。还是结合个例子来看看，不然比较模糊

   function foo(){
      var a = 1
      function bar(){
         console.lgo(a)
      }
      bar()
   }
   foo()
